## 一、技术选型对比

### 1.1 Web 框架选型

| 方案 | 技术栈 | 优点 | 缺点 | 推荐度 |
|------|--------|------|------|--------|
| **方案 A（当前）** | FastAPI + Jinja2 | 简单、异步、自动生成 API 文档、无需前端构建 | SSR 体验略差 | ⭐⭐⭐⭐⭐ **推荐** |
| 方案 B | Flask + Jinja2 | 简单、生态成熟 | 同步阻塞、无自动 API 文档 | ⭐⭐⭐ |
| 方案 C | Node.js + React | 现代、交互体验好 | 需要前端构建、复杂度高、维护成本高 | ⭐⭐ |

**决策**: ✅ **方案 A（FastAPI + Jinja2）**  
**理由**:
1. 项目核心是**内容聚合 + AI 改写**，非高频交互应用
2. Server-Side Rendering 足够满足需求
3. FastAPI 异步特性适合处理 LLM API 调用（长时间 I/O）
4. Jinja2 模板简单直接，无需 Node.js 构建流程
5. 避免引入前端技术栈，降低维护成本

### 1.2 数据库选型

| 方案 | 技术栈 | 优点 | 缺点 | 推荐度 |
|------|--------|------|------|--------|
| **方案 A（当前）** | SQLite | 零配置、单文件、适合单机部署 | 不支持高并发写入 | ⭐⭐⭐⭐ **推荐** |
| 方案 B | PostgreSQL | 功能强大、支持高并发 | 需要单独安装、配置复杂 | ⭐⭐⭐ |
| 方案 C | MySQL | 生态成熟、性能稳定 | 需要单独安装、配置复杂 | ⭐⭐⭐ |

**决策**: ✅ **方案 A（SQLite）**  
**理由**:
1. 项目是**单机部署**（非分布式系统）
2. 主要场景：**读取 > 写入**（采集后的文章存储）
3. SQLite 性能足够（实测：10,000 篇文章无压力）
4. 零配置，直接随项目分发
5. 支持通过 `shared/auth` 模块复用用户认证表

### 1.5 LLM 客户端选型

| 方案 | 技术栈 | 优点 | 缺点 | 推荐度 |
|------|--------|------|------|--------|
| **方案 A（当前）** | 自研 `llm_client.py` | 统一接口、支持多模型、内置重试 | 需要手动维护 | ⭐⭐⭐⭐ **推荐** |
| 方案 B | LangChain | 功能强大、生态成熟 | 抽象过度、学习成本高 | ⭐⭐⭐ |
| 方案 C | 直接调用 API | 简单直接 | 无统一接口、重复代码多 | ⭐⭐ |

**决策**: ✅ **方案 A（自研 `llm_client.py`）**  
**理由**:
1. 项目需要支持**多个 LLM 提供商**（DeepSeek、SenseNova、OpenAI 兼容接口）
2. 自研客户端可以**精确控制**重试逻辑、超时、错误处理
3. 避免引入 LangChain 的重依赖（包体积 > 200MB）
4. 统一接口方便后续扩展新模型

### 2.3 数据流图

#### 2.3.1 采集流程

```
用户配置 RSS/YouTube → 点击"开始采集" →前端 AJAX 请求 → FastAPI /api/sources/collect
   → Collector.run() → RSS/YouTube API → 原始数据（title, content, url）
   → Pipeline.process() → 过滤（敏感词+去重） → 存储到 SQLite
   → WebSocket 进度更新 → 前端进度条更新
```

#### 2.3.2 改写流程

```
用户输入文本/URL → 点击"开始改写" →前端 AJAX 请求 → FastAPI /api/rewrite
   → Pipeline.rewrite() → LanguageDetector.detect() → 非中文？
      → 是：Translator.translate() → 翻译为英文
      → 否：跳过翻译
   → Rewriter.rewrite() → LLM API 调用（DeepSeek/SenseNova）
   → 解析改写结果（标题 + 正文 + 摘要）
   → 返回 JSON → 前端显示改写结果
```

#### 2.3.3 认证流程

```
用户注册 → FastAPI /api/auth/register → 检查用户名是否存在
   → 不存在：生成 UUID + 密码哈希（SHA-256）→ 存储到 user.db
   → 返回 JWT Access Token（7 天有效期）

用户登录 → FastAPI /api/auth/login → 验证用户名+密码
   → 正确：返回 JWT Access Token + Refresh Token（30 天）
   → 错误：返回 401

受保护 API → FastAPI 依赖注入 `get_current_user` → 验证 JWT Token
   → 有效：返回当前用户
   → 无效：返回 403
```

## 三、关键技术决策

### 3.1 为什么选 FastAPI？

**决策时间**: 2026-06-07  
**决策人**: QClaw (CEO)  
**决策背景**: 需要选择一个 Web 框架用于构建内容聚合器的前端 + API。

**对比分析**:

| 维度 | FastAPI | Flask | Django |
|------|---------|-------|--------|
| 异步支持 | ✅ 原生 | ❌ 需插件 | ❌ 需插件 |
| API 文档 | ✅ 自动生成 | ❌ 需手动 | ⚠️ 需插件 |
| 性能 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐ |
| 学习成本 | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐ |
| 适合场景 | API + SSR | SSR | 全栈 |

**决策理由**:
1. **异步 I/O 是核心需求**：LLM API 调用是长时间 I/O 操作，FastAPI 的 `async/await` 原生支持，Flask 需额外插件。
2. **自动生成 API 文档**：FastAPI 自动生成 Swagger UI，方便调试和测试。
3. **类型提示 + Pydantic**：请求参数自动校验，减少手动校验代码。
4. **性能**：基于 Starlette + Pydantic，性能接近 Node.js。

**风险**:
- Jinja2 模板的 SSR 体验略差（无客户端路由、无热更新）。
- **缓解措施**：项目是工具型应用，SSR 已足够；如需更好的交互体验，可后续引入 Alpine.js（无需构建）。

### 3.2 为什么选 SQLite？

**决策时间**: 2026-06-07  
**决策人**: QClaw (CEO)  
**决策背景**: 需要选择一个数据库用于存储采集的文章和用户数据。

**对比分析**:

| 维度 | SQLite | PostgreSQL | MySQL |
|------|---------|--------------|--------|
| 配置复杂度 | ⭐⭐⭐⭐⭐（零配置） | ⭐⭐ | ⭐⭐ |
| 性能（读） | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| 性能（写） | ⭐⭐⭐（单写锁） | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| 并发支持 | ⭐⭐（适合单机） | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| 部署难度 | ⭐⭐⭐⭐⭐（单文件） | ⭐⭐ | ⭐⭐ |

**决策理由**:
1. **零配置**：SQLite 是单文件数据库，无需安装数据库服务器，直接随项目分发。
2. **适合单机部署**：项目是个人工具，非分布式系统，SQLite 足够。
3. **读多写少**：采集后的文章主要被读取（展示、改写），写入频率低（定时采集）。
4. **事务支持**：SQLite 支持 ACID 事务，保证数据一致性。

**风险**:
- **写入并发限制**：SQLite 使用单写锁，同一时间只允许一个写操作。
- **缓解措施**：
  1. 项目是单机部署，并发写入场景少（只有定时采集任务）。
  2. 如需支持高并发写入，可后续迁移到 PostgreSQL。

---

### 3.3 为什么选 YAML 配置？

**决策时间**: 2026-06-07  
**决策人**: QClaw (CEO)  
**决策背景**: 需要选择一个配置格式用于存储 API Key、代理、模型参数等配置。

**对比分析**:

| 维度 | YAML | JSON | TOML |
|------|------|------|------|
| 人类可读性 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ |
| 支持注释 | ✅ | ❌ | ✅ |
| 结构复杂度 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ |
| 解析库生态 | ⭐⭐⭐⭐⭐（PyYAML） | ⭐⭐⭐⭐⭐（内置） | ⭐⭐⭐（tomli） |

**决策理由**:
1. **支持注释**：配置文件需要频繁手动编辑，注释可以说明每个配置项的用途（例如：`# 代理地址，留空则不使用代理`）。
2. **结构清晰**：YAML 的缩进结构直观，嵌套配置易于阅读。
3. **人类友好**：YAML 支持多行字符串（`|`）、列表缩进，适合配置复杂的 LLM 提示词。

**风险**:
- **缩进敏感**：YAML 使用缩进表示层级，缩进错误会导致解析失败。
- **缓解措施**：项目提供 `config.yaml` 模板，用户直接修改值即可，无需修改结构。

---

### 3.4 为什么选无构建前端？

**决策时间**: 2026-06-07  
**决策人**: QClaw (CEO)  
**决策背景**: 需要选择前端技术栈用于构建 Web UI。

**对比分析**:

| 维度 | 无构建（Jinja2 + 原生 JS） | React + Webpack | Vue + Vite |
|------|---------------------------|-----------------|--------------|
| 构建复杂度 | ⭐⭐⭐⭐⭐（无需构建） | ⭐⭐ | ⭐⭐⭐ |
| 部署复杂度 | ⭐⭐⭐⭐⭐（直接拷贝） | ⭐⭐ | ⭐⭐⭐ |
| 交互体验 | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 维护成本 | ⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐ |
| 适合场景 | 工具型应用 | 复杂前端应用 | 现代 Web 应用 |

**决策理由**:
1. **项目定位**：项目是**工具型应用**（内容聚合 + AI 改写），非复杂前端应用（如仪表盘、编辑器）。
2. **页面数量有限**：约 10 个页面，交互逻辑简单（表单提交、进度显示、AJAX 请求）。
3. **避免构建流程**：无需引入 Node.js、Webpack、Vite 等构建工具，降低部署复杂度。
4. **Jinja2 已足够**：Jinja2 支持模板继承（`{% extends %}`）、组件复用（`{% include %}`、`{% macro %}`），可以满足需求。

**风险**:
- **代码复用略差**：原生 JS 无模块化（ES6 模块需浏览器支持），代码复用需手动拷贝。
- **缓解措施**：
  1. 项目已使用 `style-pages.css` 统一管理样式，避免 inline styles。
  2. 后续可引入 Alpine.js（无需构建），增强交互体验。

### 3.5 为什么自研 LLM 客户端？

**决策时间**: 2026-06-08  
**决策人**: QClaw (CEO)  
**决策背景**: 需要选择一个 LLM 客户端方案用于调用多个提供商的 API（DeepSeek、SenseNova、OpenAI 兼容接口）。

**对比分析**:

| 维度 | 自研 `llm_client.py` | LangChain | 直接调用 API |
|------|----------------------|------------|--------------|
| 多模型支持 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐ |
| 依赖大小 | ⭐⭐⭐⭐⭐（0 依赖） | ⭐（> 200MB） | ⭐⭐⭐⭐⭐（0 依赖） |
| 学习成本 | ⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐⭐⭐ |
| 控制精度 | ⭐⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐⭐⭐ |

**决策理由**:
1. **多提供商支持**：项目需要支持 DeepSeek、SenseNova、OpenAI 兼容接口，自研客户端可以统一接口。
2. **精确控制**：自研客户端可以精确控制重试逻辑（指数退避）、超时（30s）、错误处理（401 自动更换 Key）。
3. **避免重依赖**：LangChain 包体积 > 200MB，项目是轻量级工具，不应引入。
4. **统一接口**：自研客户端提供统一接口（`LLMClient.call()`），方便后续扩展新模型。

**风险**:
- **维护成本**：需要手动维护 `llm_client.py`，跟进各提供商的 API 变更。
- **缓解措施**：
  1. 项目已封装 `LLMClient` 类，各提供商通过 `provider` 字段区分，新增模型只需添加配置。
  2. 后续可引入 `openai` Python SDK（体积小、统一接口），替代自研客户端。
