## 一、背景与目标

### 1.1 背景

当前内容创作团队在热文采集和改写过程中存在以下痛点：
- 小红书/抖音等平台内容需手动整理，效率低
- 改写策略通用，无法按行业语境调整
- 内置策略固定，无法自定义保存
- 认证系统不统一，多项目无法复用
- 前端界面需要现代化改造
- 微信发布流程需要完善（封面管理、密码找回等）
- 多平台发布需要手动操作，效率低
- YouTube 字幕提取失败时无有效 fallback
- 缺少任务取消功能
- 代理配置硬编码，切换代理软件后需手动修改
- YouTube 采集排序固定，无法按需求调整
- 微信封面选择优先级不明确，AI 生图功能未完整接入

### 1.2 目标

- 支持链接采集（小红书/抖音）并自动转写文案
- 支持行业定向改写，提升输出专业度
- 支持自定义改写策略管理，提升复用率
- 统一认证系统，支持多项目复用
- 完善微信发布流程（封面管理、密码找回、AI 生图）
- 优化前端界面和用户体验
- 支持多平台一键发布，提升发布效率
- 优化 YouTube 字幕提取 fallback 逻辑
- 实现忘记密码功能
- 新增任务取消功能
- 优化代理配置自动检测
- **新增**：支持 YouTube 采集排序选项配置
- **新增(v1.8.0)**：完善微信封面选择优先级链 + AI 生图接入 + 默认封面存储方案

### 功能 24：YouTube 采集排序选项

#### 4.1 需求背景

当前 YouTube 数据采集时，排序方式固定（默认为播放量排序），无法根据需求调整为最新、评分或相关性排序，限制了采集灵活性。

#### 4.2 实现方式

**修改文件**：
- `web/templates/settings.html`：在数据源设置页面的 YouTube 部分增加排序选项
- `config/config.yaml`：新增 `youtube.search_order` 配置项
- `src/content_aggregator/sources/collectors/youtube_collector.py`：读取配置中的排序选项

**配置示例**：
```yaml
youtube:
  search_order: viewCount  # 可选值：date, viewCount, rating, relevance，默认为 viewCount
```

**前端界面**：
- 在 YouTube 设置区域新增下拉框：
  - 选项：最新 (date)、播放量 (viewCount)、评分 (rating)、相关性 (relevance)
  - 默认值：播放量 (viewCount)
  - 提示文字："选择 YouTube 搜索结果的排序方式"

**后端逻辑**：
- 读取配置文件中的 `youtube.search_order`
- 在调用 YouTube Data API 的 `search().list()` 时，传入 `order` 参数
- 支持的排序方式：
  - `date`：按时间最新排序
  - `viewCount`：按播放量排序
  - `rating`：按评分排序
  - `relevance`：按相关性排序（默认）

#### 4.3 技术约束

| 项目 | 说明 |
|------|------|
| YouTube Data API | 使用 v3 版本，支持 `order` 参数 |
| 配置热更新 | 修改配置后无需重启，自动生效 |
| 默认值 | 播放量 (viewCount) |

#### 4.4 API 设计

**配置读取**：
- 读取路径：`config.config.yaml` 中的 `youtube.search_order`
- 默认值：`viewCount`

**API 调用示例**：
```python
youtube.search().list(
    part="snippet",
    q=query,
    type="video",
    order=config.youtube.search_order,  # 新增参数
    maxResults=max_results
)
```

#### 4.5 验收标准

- **Given** 用户在设置页面选择"最新"
  **When** 采集 YouTube 视频
  **Then** 搜索结果按上传时间最新排序

- **Given** 用户在设置页面选择"播放量"
  **When** 采集 YouTube 视频
  **Then** 搜索结果按播放量从高到低排序

- **Given** 用户未配置排序选项
  **When** 采集 YouTube 视频
  **Then** 使用默认排序（播放量）

- **Given** 用户在设置页面选择"评分"
  **When** 采集 YouTube 视频
  **Then** 搜索结果按评分从高到低排序

- **Given** 用户在设置页面选择"相关性"
  **When** 采集 YouTube 视频
  **Then** 搜索结果按与关键词的相关性排序

### 功能 30：微信公众号封面优先级链 + AI 生图 + 默认封面存储

#### 30.1 需求背景

发布微信文章时，封面图是提升点击率的关键要素。目前：
1. 封面来源选择分散（AI 生图、自定义上传、自动提取）缺乏统一的优先级规则
2. AI 封面生图功能（`image_gen.py`）模块存在但未完整接入发布流程
3. 默认封面的存储方案（本地 vs 微信素材库）未明确

需要建立清晰的封面优先级链，让系统自动选择最合适的封面。

#### 30.2 封面优先级链（三档自动降级）

发布时按以下优先级自动匹配封面，上一级命中则跳过后续步骤：

| 优先级 | 封面来源 | 触发条件 | 前端操作 |
|--------|---------|----------|----------|
| **P1** | **AI 生成封面** | 用户在发布弹窗中点击「AI 生成封面」并成功生成 | 选择 coverSource = 'ai'，传入 `cover_id` |
| **P2** | **正文首张图片** | 无 AI 封面时，自动从文章正文 HTML 中提取首张图片 | coverSource = 'auto' 或不传 cover_id |
| **P3** | **默认封面** | 正文无图片时，使用用户在设置页上传的默认封面 | coverSource = 'auto'，fallback 到默认封面 |
| **保底** | **绿色占位图** | 以上均不可用，临时生成纯色占位图 | 不触发（极端情况兜底）|

**流程图**：

```
开始发布
  ↓
有 AI 生成封面？ ──Yes──→ 上传 AI 封面 → 发布
  ↓ No
正文有图片？ ──Yes──→ 提取首图 → 上传 → 发布
  ↓ No
有默认封面？ ──Yes──→ 上传默认封面 → 发布
  ↓ No
生成纯色占位图 → 发布
```

#### 30.4 默认封面存储方案决策

**选择方案：本地文件 + media_id 缓存（混合方案）**

| 维度 | 纯本地 | 纯微信素材库 | ✅ 推荐：本地 + 缓存 |
|------|--------|-------------|-------------------|
| 管理便捷性 | ✅ 直接文件操作| ❌ 需 API 调用 | ✅ 本地管理 + 自动上传 |
| 发布效率 | ❌ 每次需上传 | ✅ 永久 media_id | ✅ 缓存命中时秒级 |
| 存储上限 | ✅ 仅受磁盘限制 | ❌ 1000(认证)/500(未认证) | ✅ 不受限 |
| 微信依赖 | ❌ 上传失败无法发布 | ✅ 一次上传永久使用 | ⚠️ 轻度依赖，有缓存 |
| 实现复杂度 | ✅ 简单 | ⚠️ 中等 | ⚠️ 中等 |

**具体实现**：

1. **存储**：封面文件统一存放在 `data/covers/`，以 `cover_id.png` 命名
2. **默认封面**：上传到 `data/covers/default_cover.png`
3. **首次发布**：调用 `wx.upload_thumb()` 上传到微信临时素材 → 获得 `media_id`
4. **media_id 缓存**：将 `media_id` + 上传时间戳写入 `config/config.yaml` 的 `default_thumb_media_id` / `default_thumb_uploaded_at`
5. **缓存刷新**：临时素材有效期 3 天，提前 0.5 天（2.5 天后）重新上传
6. **AI 封面**：生成时写本地，发布时按需上传（不缓存，一次性使用）

**缓存策略核心代码逻辑**（已在 `wechat_router.py` 中实现）：
```python
cached_mid = cfg.get("default_thumb_media_id")
cached_at = cfg.get("default_thumb_uploaded_at", 0)
now_ts = time.time()
if cached_mid and (now_ts - cached_at) < 2.5 * 86400:
    thumb_media_id = cached_mid  # 复用缓存
else:
    thumb_media_id = upload_thumb(token, default_cover_path)
    # 写入缓存
    cfg["default_thumb_media_id"] = thumb_media_id
    cfg["default_thumb_uploaded_at"] = now_ts
```

#### 30.5 发布流程集成

**发布 API**：`POST /api/wechat/publish/{article_id}`

**请求参数**：

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `cover_id` | string | 否 | 指定封面 ID（优先级 P1：AI 生图或自定义上传）|
| `theme` | string | 否 | 排版主题 |
| `author` | string | 否 | 作者名 |

**后端执行流程**（修改 `wechat_router.py` 中 `api_publish_article()`）：

```
1. 排版转换（HTML + 图片提取）
2. 封面选择（按优先级链）：
   a. 如果 cover_id 有值 → 使用指定封面 → 上传到微信 → 获取 media_id
   b. 如果 cover_id 为空，从正文提取首图 → 下载 → 上传到微信
   c. 如果无正文图片，使用默认封面 → 检查缓存 media_id → 上传/复用
   d. 如果无默认封面，生成绿色占位图 → 上传
3. 调用 WeChat create_draft()
4. 返回 drafts/{media_id}
```

#### 30.6 技术约束

| 项目 | 说明 |
|------|------|
| 图片格式 | 仅支持 JPG/PNG/GIF/WebP（上传校验）|
| 文件大小 | 上传 ≤ 10MB |
| media_id 缓存周期 | 2.5 天（临时素材有效期内提前刷新）|
| AI 生图超时 | 每个提供商 120 秒，失败自动切下家 |
| 默认封面存储 | `data/covers/default_cover.png` |
| AI 封面存储 | `data/covers/{uuid12}.png` |
| 正文图片提取 | 正则 `<img[^>]+src=["']([^"']+)["']`，取第一个匹配 |

#### 30.7 验收标准

- **Given** 用户在发布弹窗中成功生成 AI 封面
  **When** 点击发布
  **Then** 封面使用 AI 生成的图片

- **Given** 无 AI 封面，正文包含图片
  **When** 点击发布
  **Then** 封面使用正文首张图片（自动下载并上传）

- **Given** 无 AI 封面且正文无图片，但设置了默认封面
  **When** 点击发布
  **Then** 封面使用默认封面（优先复用缓存 media_id）

- **Given** 无 AI 封面、正文无图、无默认封面
  **When** 点击发布
  **Then** 使用绿色占位图作为封面，发布不失败

- **Given** 默认封面已上传且 media_id 在有效期
  **When** 连续第二次发布
  **Then** 不重复上传，复用缓存的 media_id

- **Given** 默认封面 media_id 超过 2.5 天
  **When** 发布
  **Then** 重新上传并更新缓存
