## 一、背景与目标

### 1.1 背景

当前内容创作团队在热文采集和改写过程中存在以下痛点：
- 小红书/抖音等平台内容需手动整理，效率低
- 改写策略通用，无法按行业语境调整
- 内置策略固定，无法自定义保存
- 认证系统不统一，多项目无法复用
- 前端界面需要现代化改造
- 微信发布流程需要完善（封面管理、密码找回等）
- 多平台发布需要手动操作，效率低
- YouTube 字幕提取失败时无有效 fallback
- 缺少任务取消功能
- 代理配置硬编码，切换代理软件后需手动修改

### 1.2 目标

- 支持链接采集（小红书/抖音）并自动转写文案
- 支持行业定向改写，提升输出专业度
- 支持自定义改写策略管理，提升复用率
- 统一认证系统，支持多项目复用
- 完善微信发布流程（封面管理、密码找回）
- 优化前端界面和用户体验
- 支持多平台一键发布，提升发布效率
- 优化 YouTube 字幕提取 fallback 逻辑
- 实现忘记密码功能
- 新增任务取消功能
- 优化代理配置自动检测

### 1.3 成功指标

| 指标 | 目标值 |
|------|--------|
| 内容产出效率 | 提升 50%（篇/小时）|
| 改写质量满意度 | ≥ 85% |
| 策略复用率 | ≥ 60% |
| 用户认证成功率 | ≥ 99% |
| 微信发布成功率 | ≥ 95% |
| 多平台发布成功率 | ≥ 90% |
| YouTube 字幕提取成功率 | ≥ 95% |

## 四、功能详情

### 功能 18：多平台一键发布

#### 4.1 需求背景

当前热文采集改写后，需要手动发布到多个平台（微信公众号、知乎专栏、掘金、CSDN 等），操作繁琐，效率低。

#### 4.2 实现方式

**新增文件**：
- `web/templates/multi_publish.html`：多平台发布页面
- `tools/multi_publish/publisher.py`：多平台发布工具

**支持平台**：
- 微信公众号（已有）
- 知乎专栏
- 掘金
- CSDN
- 自定义平台（Webhook）

**发布流程**：
1. 用户选择要发布的文章
2. 选择目标平台（可多选）
3. 配置各平台发布参数（标题、标签、分类等）
4. 一键发布到所有选中平台
5. 显示发布结果（成功/失败）

#### 4.3 技术约束

| 项目 | 说明 |
|------|------|
| 平台 API | 需要各平台 API Key |
| 发布频率限制 | 遵守各平台 API 调用频率限制 |
| 错误处理 | 单个平台发布失败不影响其他平台 |

#### 4.4 API 设计

| 端点 | 方法 | 功能 |
|------|------|------|
| `/api/publish/multi` | POST | 多平台发布 |
| `/api/publish/status/{task_id}` | GET | 查询发布状态 |

#### 4.5 验收标准

- **Given** 用户选择一篇文章并选择多个目标平台
  **When** 点击「一键发布」
  **Then** 文章发布到所有选中平台，显示发布结果

- **Given** 某个平台发布失败
  **When** 查看发布结果
  **Then** 显示失败原因，其他平台发布成功

### 功能 19：YouTube 字幕提取 Fallback

#### 4.1 需求背景

当前 YouTube 字幕提取逻辑中，当首选语言（zh/en）都失败时，直接回退到视频描述，导致字幕内容缺失。

#### 4.2 实现方式

**修改文件**：`src/content_aggregator/sources/collectors/youtube_collector.py`

**Fallback 逻辑**：
1. 尝试提取中文子字幕
2. 尝试提取英文字幕
3. 尝试提取任意可用字幕（不限制语言）
4. 所有字幕提取失败，回退到视频描述

#### 4.3 技术约束

| 项目 | 说明 |
|------|------|
| 字幕格式 | 支持 srt、vtt、json3 |
| 语言检测 | 使用 YouTube API 返回的字幕列表 |

#### 4.4 验收标准

- **Given** YouTube 视频有日文字幕，无中英文字幕
  **When** 提取字幕
  **Then** 成功提取日文字幕

- **Given** YouTube 视频无任何字幕
  **When** 提取字幕
  **Then** 回退到视频描述

### 功能 20：忘记密码功能实现

#### 4.1 需求背景

当前用户忘记密码时无法自行重置，需要联系管理员手动重置，体验差。

**注意**：此功能在 INBOX 中标记为 P1，但在 v1.3.0 PRD 中已列为功能 8（密码找回功能），状态为"待实现"。本次 v1.5.0 将重新梳理并实现。

#### 4.2 实现方式

**新增页面**：
- `web/auth/forgot_password.html`：输入注册邮箱
- `web/auth/reset_password.html`：输入新密码（带 token 验证）

**后端新增端点**：
- `POST /api/auth/forgot-password`：发送密码重置邮件
- `POST /api/auth/reset-password`：重置密码（验证 token）

**业务流程**：
1. 用户点击「忘记密码」
2. 输入注册邮箱，系统发送重置链接（含 JWT reset token）
3. 用户点击链接，跳转到重置密码页面
4. 输入新密码，提交
5. 系统验证 token（1h 有效期），更新密码

#### 4.3 技术约束

| 项目 | 说明 |
|------|------|
| Token 类型 | JWT reset token |
| Token 有效期 | 1h（3600 秒）|
| 邮件发送 | 使用 SMTP 或邮件服务 API |
| Token 存储 | 不存储（JWT 自包含）|

#### 4.4 验收标准

- **Given** 用户输入注册邮箱
  **When** 点击「发送重置邮件」
  **Then** 收到重置邮件（≤ 1min）

- **Given** 用户点击重置链接（token 有效）
  **When** 输入新密码，提交
  **Then** 密码重置成功，可使用新密码登录

- **Given** 用户点击重置链接（token 过期）
  **When** 输入新密码，提交
  **Then** 提示"重置链接已过期，请重新申请"

### 功能 21：任务取消功能

#### 4.1 需求背景

当前任务列表页面缺少取消功能，一旦任务启动（pending/running），无法取消，只能等待完成或手动杀进程。

#### 4.2 实现方式

**前端修改**：`web/templates/tasks.html`

**新增按钮**：为 pending/running 状态的任务添加「取消」按钮

**后端修改**：`web/server.py`、`TaskManager`

**新增方法**：`TaskManager.cancel(task_id)`

**新增端点**：`DELETE /api/tasks/{task_id}`

**取消逻辑**：
1. 前端发送取消请求
2. 后端标记任务状态为 `cancelling`
3. 任务执行逻辑检查取消标志，优雅退出
4. 标记任务状态为 `cancelled`

#### 4.3 技术约束

| 项目 | 说明 |
|------|------|
| 任务类型 | 支持采集任务、改写任务 |
| 取消方式 | 优雅退出（不强制杀进程）|
| 状态同步 | 前端轮询或 WebSocket 推送 |

#### 4.4 API 设计

| 端点 | 方法 | 功能 |
|------|------|------|
| `/api/tasks/{task_id}` | DELETE | 取消任务 |

#### 4.5 验收标准

- **Given** 任务状态为 pending 或 running
  **When** 用户点击「取消」按钮
  **Then** 任务状态变为 cancelling，随后变为 cancelled

- **Given** 任务状态为 completed 或 failed
  **When** 查看任务列表
  **Then** 不显示「取消」按钮

### 功能 23：代理配置自动检测优化

#### 4.1 需求背景

当前代理端口配置硬编码为单个端口（7890 或 12334），当代理软件切换后，配置失效，需要手动修改。

#### 4.2 实现方式

**修改文件**：
- `config/config.yaml`：支持配置多个代理端口
- `team/scripts/auto-proxy-git.ps1`：优化代理检测逻辑

**检测逻辑**：
1. 读取配置的代理端口列表（如 [7890, 12334, 7892]）
2. 按顺序检测端口是否可用（TCP 连接测试）
3. 使用第一个可用端口
4. 所有端口都不可用，提示用户

**配置示例**：
```yaml
proxy:
  ports:
    - 7892
    - 7890
    - 12334
  check_timeout: 3  # 每个端口检测超时时间（秒）
```

#### 4.3 技术约束

| 项目 | 说明 |
|------|------|
| 检测方式 | TCP 连接测试（System.Net.Sockets.TcpClient）|
| 检测超时 | 3 秒/端口 |
| 配置热更新 | 支持修改配置后自动重新检测 |

#### 4.4 验收标准

- **Given** 代理软件使用端口 7892
  **When** 系统启动
  **Then** 自动检测到 7892 可用，使用此端口

- **Given** 代理软件从 7890 切换到 12334
  **When** 系统重新检测代理配置
  **Then** 自动切换到 12334

- **Given** 所有配置的端口都不可用
  **When** 系统启动
  **Then** 提示用户"未检测到可用代理端口，请检查代理软件是否启动"
