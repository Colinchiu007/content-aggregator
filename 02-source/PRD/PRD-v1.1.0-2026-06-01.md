## 一、背景与目标

### 1.1 背景

当前内容创作团队在热文采集和改写过程中存在以下痛点：
- 小红书/抖音等平台内容需手动整理，效率低
- 改写策略通用，无法按行业语境调整
- 内置策略固定，无法自定义保存

### 1.2 目标

- 支持链接采集（小红书/抖音）并自动转写文案
- 支持行业定向改写，提升输出专业度
- 支持自定义改写策略管理，提升复用率

### 1.3 成功指标

| 指标 | 目标值 |
|------|--------|
| 内容产出效率 | 提升 50%（篇/小时）|
| 改写质量满意度 | ≥ 85% |
| 策略复用率 | ≥ 60% |

## 2. 功能 1：链接采集（小红书/抖音）

### 2.1 需求背景

用户有小红书/抖音链接，希望将视频内容转为文字文案，用于后续改写分发。当前系统无此入口。

### 2.2 页面设计

**左侧菜单新增**：`链接采集`（icon: link）

**链接采集页面** (`/collect-link`)：

```
┌─────────────────────────────────────┐
│  链接采集                            │
├─────────────────────────────────────┤
│                                       │
│  粘贴链接                             │
│  ┌─────────────────────────────────┐ │
│  │ https://www.xiaohongshu.com/... │ │
│  └─────────────────────────────────┘ │
│                                       │
│  目标平台  [小红书 ▼]  [抖音 ▼]       │
│                                       │
│  [开始采集]                           │
│                                       │
│  ── 采集结果 ─────────────────────   │
│  标题：...                            │
│  原文案：...                          │
│  转写文案：...                        │
│  [查看详情] [改写] [导出]             │
└─────────────────────────────────────┘
```

### 2.3 业务逻辑

```
用户输入链接
    ↓
识别平台（小红书 / 抖音 / 自动识别）
    ↓
调用对应 Collector 获取视频/帖子元数据
    ↓
若有语音 → 调用 ASR 转写文字
    ↓
若无语音（纯图文）→ 直接提取正文
    ↓
展示结果（文章详情样式）
    ↓
支持后续操作：改写 / 导出 / 保存
```

### 2.4 字段与规则

| 字段 | 类型 | 校验 | 来源 |
|------|------|------|------|
| `source_url` | String | 必须有效 URL | 用户输入 |
| `platform` | Enum | xiaohongshu / douyin / auto | 用户选择或自动识别 |
| `media_type` | Enum | video / image_text | Collector 返回 |
| `original_text` | Text | 图文帖原文案 | Collector 返回 |
| `transcribed_text` | Text | ASR 转写结果 | ASR 服务返回 |
| `final_text` | Text | 合并后文案 | 系统合成 |

**平台识别规则**：
- URL 含 `xiaohongshu.com` / `xhslink.com` → 小红书
- URL 含 `douyin.com` / `iesdouyin.com` → 抖音
- 其他 → 报错提示不支持

### 2.5 技术约束

| 项目 | 说明 |
|------|------|
| ASR 服务 | 优先用 DeepSeek/OpenAI Whisper API；若无，提示用户手动上传字幕 |
| 小红书 Cookie | 需在 config.yaml 配置 `sources.xiaohongshu.cookie`，失效时提示重新配置 |
| 抖音 Cookie | 同小红书，需 `sources.douyin.cookie` |
| 超时 | 单条采集 ≤ 60s，超时展示错误状态 |

### 2.6 文章详情展现

复用现有 `/articles/{id}` 详情页样式，新增「链接采集」来源类型：

- 顶部标注来源平台 icon + 原始链接
- 展示区：原文案（若有）+ ASR 转写文案
- 操作栏：改写（跳转 `/compose` 并带入文案）、导出、保存

## 3. 功能 2：文章改写 - 行业定向

### 3.1 需求背景

当前改写策略是通用的，不支持按行业语境调整。用户希望指定目标行业（如"美妆"、"金融科技"），让 AI 按行业术语和读者偏好改写。

### 3.2 交互变更

**修改页面**：`/compose`（手动输入改写）、API `/api/rewrite`

**新增字段**：`目标行业`（文本输入框，可选）

```
┌─────────────────────────────────────┐
│  手动改写                            │
├─────────────────────────────────────┤
│  输入文案／粘贴链接                  │
│  ┌─────────────────────────────────┐ │
│  │ ...                             │ │
│  └─────────────────────────────────┘ │
│                                       │
│  改写策略  [深度改写 ▼]               │
│  目标行业  [           ]  (可选)      │
│             ↑ 用户输入，如"美妆"      │
│                                       │
│  [开始改写]                           │
└─────────────────────────────────────┘
```

### 3.3 Prompt 调整

在 `RewriteProcessor` 的 system prompt 中，若用户填写了行业，追加：

```
目标行业：{industry}
请基于该行业的读者语境、专业术语和表达习惯进行改写。
```

**示例**：
- 行业=美妆 → 使用"成分"、"肤感"、"上脸"等术语
- 行业=金融科技 → 使用"年化"、"回撤"、"仓位"等术语

### 3.4 字段与规则

| 字段 | 类型 | 校验 | 来源 |
|------|------|------|------|
| `industry` | String | 最大 20 字符，可选 | 用户输入 |
| `strategy` | Enum | 已有 6 种策略 | 用户选择 |

**边界条件**：
- 未填写行业 → 通用改写（保持现有行为，向后兼容）
- 行业名称随意输入，不做枚举限制（支持长尾行业）

## 4. 功能 3：改写策略管理

### 4.1 需求背景

用户有多种改写需求（如"小红书风格"、"知乎长文"、"微信公众号"），当前只能选内置 6 种策略，无法自定义。需要支持用户保存多份自定义策略，并设默认。

### 4.2 页面设计

**左侧菜单新增**：`改写策略`（icon: settings / sparkles）

**策略管理页面** (`/rewrite-strategies`)：

```
┌─────────────────────────────────────┐
│  改写策略管理                        │
├─────────────────────────────────────┤
│  [+ 新建策略]                         │
│                                       │
│  策略列表：                           │
│  ┌─────────────────────────────────┐ │
│  │ ★ 默认策略 (深度改写)   [编辑][删除]│ │
│  │ 小红书爆款风格        [编辑][删除]│ │
│  │ 知乎专业长文          [编辑][删除]│ │
│  └─────────────────────────────────┘ │
└─────────────────────────────────────┘
```

**新建/编辑策略弹窗**：

```
┌─────────────────────────────────────┐
│  新建策略                            │
├─────────────────────────────────────┤
│  策略名称 *                          │
│  ┌─────────────────────────────────┐ │
│  │ 小红书爆款风格                   │ │
│  └─────────────────────────────────┘ │
│                                       │
│  改写需求描述 *                       │
│  ┌─────────────────────────────────┐ │
│  │ 将文案改写为小红书爆款风格，     │ │
│  │ 要求：口语化、带 emoji、         │ │
│  │ 每段不超过 3 行...              │ │
│  └─────────────────────────────────┘ │
│                                       │
│  ☑ 设为默认策略                       │
│                                       │
│  [保存]  [取消]                       │
└─────────────────────────────────────┘
```

### 4.3 数据模型

**新增表 `rewrite_strategies`**：

```sql
CREATE TABLE rewrite_strategies (
    id TEXT PRIMARY KEY,
    name TEXT NOT NULL,
    description TEXT NOT NULL,      -- 用户自定义的改写需求描述
    is_default INTEGER DEFAULT 0,   -- 0=否 1=是，全局只能有 1 条默认
    created_at TEXT DEFAULT CURRENT_TIMESTAMP,
    updated_at TEXT DEFAULT CURRENT_TIMESTAMP
);
```

### 4.4 业务逻辑

```
用户选择策略（Compose 页面下拉框）
    ↓
若选择「自定义策略」→ 读取 rewrite_strategies 表
    ↓ 若选择「内置策略」→ 读取现有 6 种策略
    ↓
将策略描述注入 RewriteProcessor 的 system prompt
    ↓
执行改写
```

**默认策略逻辑**：
- 新建策略时勾选「设为默认」→ 先将其他策略 `is_default=0`，再设当前为 `1`
- Compose 页面默认选中默认策略
- 删除默认策略 → 提示先设置新默认

### 4.5 API 变更

| 端点 | 方法 | 功能 |
|------|------|------|
| `/api/rewrite-strategies` | GET | 获取策略列表（内置 + 自定义）|
| `/api/rewrite-strategies` | POST | 新建自定义策略 |
| `/api/rewrite-strategies/{id}` | PATCH | 更新策略 |
| `/api/rewrite-strategies/{id}` | DELETE | 删除策略 |
| `/api/rewrite` | POST | 新增 `strategy_id` 参数（支持传入自定义策略 ID）|

## 5. 功能 4：微信公众号文章采集

### 5.1 需求背景

用户需要采集微信公众号历史文章，用于内容参考和改写。系统已有 WeChatCollector 实现，但未写入 PRD。

### 5.2 实现方式

**已有实现**：`src/content_aggregator/sources/collectors/wechat_collector.py`（WeChatCollector）

支持两种采集方式：
1. **通过 biz 参数获取历史消息**：`https://mp.weixin.qq.com/cgi-bin/appmsg`（需 Cookie）
2. **通过公众号名称搜索**：使用第三方 API（如新榜 API，需 api_key）

### 5.3 配置要求

| 配置项 | 说明 | 必填 |
|--------|------|------|
| `sources.wechat.api_key` | 第三方 API Key（新榜等）| 否 |
| `sources.wechat.name` | 公众号名称（搜索用）| 否 |
| `sources.wechat.biz` | 公众号 biz 参数（直接获取历史用）| 否 |
| `sources.wechat.cookie` | 微信公众号 Cookie（直接获取历史用）| 否 |

**注意**：`api_key` 和 `name` 至少提供一个，否则采集器抛出 `EnvironmentError`。

### 5.4 业务逻辑

```
用户输入公众号名称 / biz 参数
    ↓
WeChatCollector._fetch(biz, name, keyword, max_results)
    ↓
方式1：通过 biz 获取历史消息（需 Cookie）
方式2：通过名称搜索公众号（需 api_key）
    ↓
返回文章列表（标题、链接、发布时间）
    ↓
存入 articles 列表
```

### 5.5 API 端点（已有）

| 端点 | 方法 | 功能 |
|------|------|------|
| `/api/collect/wechat` | POST | 触发微信公众号文章采集 |

**请求体**：
```json
{
  "name": "公众号名称",
  "biz": "MzA3...",
  "keyword": "关键词（可选）",
  "max_results": 20
}
```

**响应**：同 `/api/collect/youtube`（返回采集结果列表）

### 5.6 验收标准

- **Given** 用户已配置 `sources.wechat.name`
  **When** 触发微信采集
  **Then** 返回公众号文章列表（≤ 60s）

- **Given** 用户未配置 `name` 或 `api_key`
  **When** 触发微信采集
  **Then** 提示"微信公众号采集未配置，请在设置中配置 sources.wechat.name 或 sources.wechat.api_key"

## 7. 功能 6：微信公众号草稿发布

### 7.1 需求背景

用户改写完内容后，希望直接发布到微信公众号，完成"采集 → 改写 → 发布"全流程闭环。

### 7.2 实现方式

**已有实现**：`src/content_aggregator/web/server.py` 中有 `/api/publish/wechat` 端点（待确认权限）。

**发布流程**：
1. 用户点击"发布到微信公众号"
2. 系统调用微信公众号 API（`cgi-bin/draft/add`）创建草稿
3. 用户手动在微信公众平台发布草稿（或系统自动发布，需权限）

### 7.3 权限要求

| 权限 | 说明 | 获取方式 |
|------|------|----------|
| 公众号认证 | 个人订阅号无法使用发布 API | 认证为服务号 / 企业号 |
| API 权限 | 需在微信公众平台申请 | 登录 mp.weixin.qq.com → 设置 → 基本配置 |

**当前状态**：等待用户确认账号权限（是否能调用发布 API）。

### 7.4 业务逻辑

```
用户点击"发布到微信公众号"
    ↓
检查微信公众号 Access Token（是否有效）
    ↓
调用 `cgi-bin/draft/add` 创建草稿
    ↓
返回草稿 media_id
    ↓
（可选）调用 `cgi-bin/freepublish/submit` 正式发布
    ↓
返回发布状态
```

### 7.5 API 设计（待确认权限后完善）

| 端点 | 方法 | 功能 |
|------|------|------|
| `/api/publish/wechat` | POST | 发布到微信公众号（创建草稿或正式发布）|

**请求体**：
```json
{
  "article_id": "uuid-xxx",
  "publish_type": "draft"
}
```

**响应**：
```json
{
  "success": true,
  "media_id": "xxx",
  "publish_type": "draft",
  "message": "草稿已创建，请在微信公众平台发布"
}
```

### 7.6 验收标准（待权限确认后制定）

- **Given** 用户已配置微信公众号 Access Token
  **When** 点击"发布到微信公众号"
  **Then** 草稿创建成功，返回 media_id

- **Given** 用户未配置 Access Token
  **When** 点击"发布到微信公众号"
  **Then** 提示"请先在设置中配置微信公众号 Access Token"
